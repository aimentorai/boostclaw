## Vision

### Company North Star
Build AI digital labor systems that directly replace marketing growth teams. The team responsible for this builds "data employees" (数据员工) on top of OpenClaw.

### 10x Check
Don't build AI assistants that help people do marketing. Build the system where a business owner delegates an entire marketing function to an AI employee, and it just happens. The 10x is not "better chat." The 10x is "I don't need to chat at all except to say yes/no."

### Platonic Ideal
The user experience feels like managing a talented junior marketer. You give them a goal, they break it into tasks, execute them, surface decisions when uncertain, and show you results. You never see system prompts, skills, or agent configs. You see: tasks, progress, outputs, and results. The employee works 24/7 across all your channels, never forgets a deadline, and gets better over time.

## Implementation Approach

**Chosen: LLM-as-Orchestrator + OpenClaw Native**

Don't build a Task Engine. Use what OpenClaw already has: LLM function calling for orchestration, Skills for tool execution, Cron for scheduling, and Cron Run Logs for execution history.

## Key Technical Decisions

### 1. Plugin + Skill 混合架构

每个 AI 员工由一个 **Plugin**（运行时工具）+ 捆绑的 **Skill**（工作流指南）+ **配置文件** 三层定义。

#### a) Plugin（运行时工具层）- 注册到 OpenClaw Gateway

Plugin 通过 `registerTool()` 注册原生 Tools，LLM 通过 Function Calling 直接调用。
对比纯 SKILL.md 方案：**确定性代码执行 > LLM 理解指令后构造 HTTP 请求**。

```
resources/openclaw-plugins/sparkboost/
  ├── openclaw.plugin.json          # Plugin 清单
  ├── dist/index.js                 # 编译后的 Plugin 代码
  ├── skills/                       # 捆绑的 Skills
  │   └── tiktok-publisher/
  │       └── SKILL.md              # 工作流指南
  └── package.json
```

Plugin 入口：

```typescript
export default definePluginEntry({
  id: "sparkboost",
  register(api) {
    // 注册 Tools：LLM 通过 Function Calling 直接调用
    api.registerTool({
      name: "sparkboost_snapshot",
      description: "查看所有账号状态、待发布视频、今日日程",
      parameters: { type: "object", properties: {} },
      async execute() {
        // 聚合查询多个 API，返回完整业务状态
        return wrapTrustBoundary(await fetchAllStatus());
      },
    });

    api.registerTool({
      name: "sparkboost_create_record",
      description: "创建 TikTok 视频发布记录",
      parameters: {
        type: "object",
        properties: {
          account: { type: "string", description: "店铺账号 ID" },
          title: { type: "string", description: "视频标题" },
          video_path: { type: "string", description: "视频文件路径" },
        },
        required: ["account", "title", "video_path"],
      },
      async execute(params) {
        const result = await sparkboostAPI.createRecord(params);
        return wrapTrustBoundary(result);
      },
    });
    // ... upload_file, check_status, publish 同理

    // 注册 Hook：强制 Trust Boundary
    api.on("after_tool_call", (ctx) => {
      if (ctx.toolName.startsWith("sparkboost_")) {
        ctx.result = wrapTrustBoundary(ctx.result, ctx.toolName);
      }
    });
  },
});
```

**Tool 三分类**（参考 GStack browse 的 READ/WRITE/META 模式）：

| 类别 | Tool 名称 | 说明 | 信任级别 |
|------|-----------|------|----------|
| **Query** | `sparkboost_snapshot` | 聚合查询：账号状态+视频+日程 | 只读，自动执行 |
| **Query** | `sparkboost_check_status` | 查询视频处理状态 | 只读，自动执行 |
| **Query** | `sparkboost_list_accounts` | 列出所有店铺及状态 | 只读，自动执行 |
| **Operate** | `sparkboost_create_record` | 创建视频记录（JSON Schema 校验参数） | 有副作用 |
| **Operate** | `sparkboost_upload_file` | 上传视频文件 | 有副作用 |
| **Operate** | `sparkboost_publish` | 定时发布视频 | 有副作用 |

**Plugin vs 纯 Skill 对比**：

| 方面 | 纯 SKILL.md | Plugin + registerTool |
|------|------------|----------------------|
| 参数校验 | LLM 理解文档后自行构造请求 | JSON Schema 自动校验，不可能传错 |
| Trust Boundary | 靠 LLM 自觉包裹 | Hook 强制包裹，确定性保证 |
| 错误处理 | LLM 解析 HTTP 响应码 | 结构化错误，确定性处理 |
| 依赖管理 | 无 | Plugin 自带 node_modules |
| 安装方式 | 手动复制到 skills 目录 | 随应用打包，自动安装到 extensions |

**Plugin 安装和部署**（和 DingTalk/WeCom 相同模式）：

```
开发: resources/openclaw-plugins/sparkboost/
  ↓ bun build
构建: build/openclaw-plugins/sparkboost/
  ↓ after-pack.cjs (Electron 打包)
分发: BoostClaw.app/.../openclaw-plugins/sparkboost/
  ↓ 用户启动 BoostClaw，ensurePluginInstalled() 自动安装
运行: ~/.openclaw/extensions/sparkboost/
  ↓ Gateway 启动时加载 Plugin
可用: registerTool() 注册的 Tools 对所有 Agent 可用
```

v1 随应用打包（零配置）。v2 可发布到 ClawHub 供第三方使用。

**Trust Boundary（信任边界）**：所有外部 API 返回值用标记包裹，防止 prompt injection：

```
--- BEGIN EXTERNAL API RESPONSE (source: sparkboost /tiktok/list) ---
{"status": "ok", "data": [...]}
--- END EXTERNAL API RESPONSE ---
```

由 Plugin 的 `after_tool_call` Hook 强制执行，不依赖 LLM。

#### b) SOUL.md（工作流指南）- Plugin 捆绑的 Skill

SOUL.md 不再教 LLM 如何调用 API（那是 Plugin Tool 的工作），只专注于工作流编排和决策边界。

```markdown
# TikTok 视频发布专家

## 工作原则
1. 行动前先 snapshot：了解全局状态再动手
2. 遇到异常不静默：报告问题，不要掩盖失败
3. 不确定时 handoff：把决定权交给用户
4. 分批执行：每次处理 2-3 个账号，汇报进度后继续

## 工作流程
当用户要求发布视频时：
1. snapshot → 查看当前账号状态和待发布视频
2. 确认参数（视频文件、目标账号、发布时间），缺少的参数要追问
3. 为每个视频生成吸引人的 TikTok 标题（LLM 推理，不调用工具）
4. 为每个账号调用 create-record（每次 2-3 个）
5. 为每条记录调用 upload-file（每次 2-3 个）
6. 调用 check-status 等待视频处理完成
7. 按用户指定的时间间隔调用 publish

## Handoff 决策边界
以下情况必须 handoff 给用户：
- 账号状态异常（suspended/banned）→ 不能自动跳过
- 上传失败 2 次 → 不能无限重试
- 标题/文案可能违规 → 让用户确认
- 涉及付费操作 → 需要用户批准

以下情况可以自动处理：
- 视频还在处理中 → 继续等待（最多 60 分钟）
- 单次上传失败 → 重试一次
- 网络超时 → 重试一次

## 默认值
- 发布间隔: 30 分钟
- 标题风格: 口语化，带热门标签
- 每批处理: 2-3 个账号
```

关键区别：SOUL.md 明确定义了 **auto-do / handoff / never-do** 三种决策类别，和 GStack 的 document-release 一样。LLM 不需要在灰色地带猜测。

### 2. LLM 即编排器

LLM 通过 Function Calling 编排混合步骤（LLM 推理 + Plugin Tool 调用），完整流程：

```
用户: "帮我把这5个视频发到所有店铺"

Step 1: Tool 调用 → 查看全局状态
  sparkboost_snapshot()
  → 返回: 3个账号正常(shop_A, shop_B, shop_C)，1个暂停(shop_D)，0个待发布
  → Plugin Hook 自动包裹 Trust boundary

Step 2: LLM 推理 → 生成标题（不调用工具）
  → 输出: ["夏季爆款！这款T恤太香了", ...]

Step 3: LLM 决策 → handoff 通知异常
  发现 shop_D 暂停 → 向用户报告 "shop_D 账号已暂停，是否跳过？"
  用户: "跳过"

Step 4: Tool 调用 → 批量创建记录（Function Calling 并行）
  sparkboost_create_record({account: "shop_A", title: "...", video_path: "/tmp/v1.mp4"})
  sparkboost_create_record({account: "shop_B", title: "...", video_path: "/tmp/v1.mp4"})
  sparkboost_create_record({account: "shop_C", title: "...", video_path: "/tmp/v1.mp4"})
  → 返回: [record_id_1, record_id_2, record_id_3]

Step 5: Tool 调用 → 批量上传
  sparkboost_upload_file({record_id: 1, file: "/tmp/v1.mp4"})
  sparkboost_upload_file({record_id: 2, file: "/tmp/v1.mp4"})
  sparkboost_upload_file({record_id: 3, file: "/tmp/v1.mp4"})
  → shop_C 失败 → Tool 内部自动重试一次 → 仍然失败
  → LLM 向用户报告 "shop_C 上传失败 2 次：文件超过 500MB 限制"
  用户: "跳过 shop_C"

Step 6: Tool 调用 → 检查状态
  sparkboost_check_status({ids: [1, 2]})
  → 返回: [ready, ready]

Step 7: Tool 调用 → 批量发布
  sparkboost_publish({record_id: 1, schedule: "2024-01-15T08:00+08:00"})
  sparkboost_publish({record_id: 2, schedule: "2024-01-15T08:30+08:00"})

Step 8: LLM 总结
  → "发布完成！2/3 店铺成功。shop_D 账号暂停，shop_C 上传失败（文件过大）。
     发布时间：8:00(shop_A), 8:30(shop_B)。需要处理失败的店铺吗？"
```

对比旧设计的改进：
1. **snapshot first**：行动前先看全局状态，避免盲目执行
2. **Plugin Tool 替代 SKILL.md 指令**：参数 JSON Schema 校验，不可能传错
3. **Trust boundary**：Plugin Hook 强制包裹，不依赖 LLM 自觉
4. **分批执行**：每次 2-3 个账号，应对 Gateway RPC 120 秒超时
5. **异常不静默**：Tool 内部重试有限次，失败后交给用户决定
6. **汇报式总结**：明确告知成功/失败/原因，不让用户猜测

### 3. 定时执行：OpenClaw Cron

OpenClaw 已有完整的 Cron 系统。LLM 通过 `cron` tool 创建定时任务：

```json
{
  "name": "每日视频发布",
  "schedule": { "kind": "cron", "expr": "0 8 * * *", "tz": "Asia/Shanghai" },
  "sessionTarget": "isolated",
  "agentId": "tiktok-publisher",
  "payload": {
    "kind": "agentTurn",
    "message": "查看昨天的选品数据，选出爆款，生成标题，发布到所有店铺"
  },
  "delivery": { "mode": "announce" }
}
```

Cron 能力（全部已内置）：
- 三种调度：一次性 (`at`)、固定间隔 (`every`)、cron 表达式 (`cron`)
- 独立会话执行（不污染主对话）
- 结果推送到渠道（Slack/Telegram/WhatsApp/Discord/Webhook）
- 自动重试（临时错误指数退避）
- Agent 绑定（指定专家执行）

BoostClaw 已有 Cron UI：`src/pages/Cron/index.tsx`

### 4. 执行历史查询：Cron Run Logs

OpenClaw 自动记录每次 Cron 执行：

```
~/.openclaw/cron/runs/<jobId>.jsonl
```

每条记录包含：status、error、duration、model、summary。

查询方式：
- LLM 调用 `cron.runs` tool 查询历史
- BoostClaw Cron UI 展示执行记录
- `electron/api/routes/cron.ts` 已有 `session-history` 接口

### 7. Integration Points with OpenClaw
- **Plugin system**: 业务 API 注册为 OpenClaw Plugin，通过 `registerTool()` 暴露为原生 Tools，随应用打包分发
- **Agent system**: 一个"营销员工" Agent，复合 SOUL.md 提供所有工作流指南。LLM 根据用户意图自动选择对应职能
- **Skills**: Plugin 捆绑的 SKILL.md 提供工作流指南，写入 Agent 系统提示
- **Cron**: 定时执行使用 OpenClaw 内置 Cron，绑定到营销员工 Agent。Payload 指定执行哪个工作流
- **Run Logs**: 执行历史使用 `~/.openclaw/cron/runs/<jobId>.jsonl`，不需要自建存储
- **Channels**: 结果通知通过 OpenClaw Cron delivery 推送到渠道

### 8. Expert-to-Employee Migration

v1 简化：不需要迁移多个专家。现有 Expert 系统处理方式：

```
On app update:
  1. 用户点击"雇佣营销员工"
  2. 创建新 Agent，注入复合 SOUL.md
  3. 用户填写 SparkBoost credentials + 默认账号列表
  4. Plugin 已随应用自动安装（ensureSparkBoostPluginInstalled）
  5. 旧 Expert 页面保留但标记为"已升级为营销员工"
```

v1.1 加新职能时：
- 复合 SOUL.md 添加新章节（如"选品分析"）
- 对应 Plugin 添加新 Tools
- 不需要创建新 Agent，不需要新模板

### 9. Testing Strategy

| Test Type | Scope | Coverage Target |
|-----------|-------|----------------|
| 手动测试 | Skill 命令：每个 Query/Operate/Manage 命令通过 Agent 对话调用 | 全部命令 |
| 手动测试 | snapshot 命令：验证返回完整的业务状态快照 | 1 个场景 |
| 手动测试 | handoff/resume：触发异常 → 验证 LLM 暂停并交给用户 → 用户处理后恢复 | 3 个异常场景 |
| 手动测试 | Trust Boundary：验证 API 返回值被正确包裹 | 1 个场景 |
| 手动测试 | 完整工作流：snapshot → 生成标题 → 创建 → 上传 → handoff(异常) → resume → 发布 | Happy path + 1 异常 |
| 手动测试 | Cron 定时执行：设置定时任务 → 等待触发 → 查看历史 | Happy path |
| E2E tests | Hire Flow：选择模板 → 填写 credentials → 创建 Agent | Critical path |
| Unit tests | employee-template.ts 模板加载和 Agent 创建 | 80%+ |
| Unit tests | employee-migration.ts Expert → Employee 迁移 | 80%+ |
| Unit tests | trust-boundary.ts API 响应包裹函数 | 100% |

**和之前方案的区别：** 之前需要测试 Task Engine、Worker Thread、MessagePort 通信、状态机等。现在这些都不存在了。LLM 编排由 OpenClaw 负责，不需要 BoostClaw 测试。新增测试重点：handoff 决策边界、trust boundary 安全性、snapshot 状态完整性。
