# MCP 接口说明

## 用途

这份说明只管一件事：什么时候该走真实 Amazon 数据，以及优先查什么。

## 默认规则

- 讲思路、讲框架、做初步排雷时，不一定需要 MCP。
- 只要开始问实时市场情况，就要优先走 `proboost-Amazon-mcp`。
- 如果 `proboost-Amazon-mcp` 当前不可用，要明确说数据源不可用，不能自己补数字。

## 这份 Skill 实际会用到哪些 MCP 接口

如果只看这份 Skill 本身，它实际依赖的 MCP 能力只有两类：

1. `tool discovery / search`
   用来确认当前会话里有没有真正加载出 `proboost-Amazon-mcp`
2. `proboost-Amazon-mcp`
   用来拿真实 Amazon 数据

也就是说，这份 Skill 的核心不是“用了很多 MCP”，而是“只在需要真实数据时，明确走 `proboost-Amazon-mcp`”。

## 这份 Skill 在 proboost-Amazon-mcp 里会用哪些接口类型

这里仍然写接口类型，不写死函数名。

原因是不同会话里，工具实际暴露出来的名字可能不完全一样，但查询意图是稳定的。

## 这个 Skill 会用到的 MCP 接口类型

这里说的是接口类型，不是写死的函数名。

原因很简单：MCP 在不同会话里，工具暴露出来的名字可能会有差异，所以运行时先发现，再调用更稳。

### 1. ASIN / Listing 查询

适合这类问题：

- 这个 ASIN 最近卖得怎么样
- 这个竞品值不值得跟
- 这个链接下面的价格和评论结构怎么样

优先拿的数据：

- 销量或单量估算
- 销售额估算
- 价格
- BSR
- 评论数和评分
- 卖家数量
- 变体数量

### 2. 关键词需求查询

适合这类问题：

- 这个关键词现在还能不能做
- 这个词是不是太卷
- 这个词下面有没有需求

优先拿的数据：

- 关键词需求量
- 对应主流产品价格带
- 竞争强度相关信号
- 关键词趋势变化

### 3. 类目 / 子类目趋势查询

适合这类问题：

- 这个类目最近是不是还在涨
- 这个小类值不值得继续看
- 这个方向是稳定盘还是下滑盘

优先拿的数据：

- 类目趋势
- 子类目趋势
- 头部产品变化
- 价格带变化
- 竞争变化

### 4. 趋势窗口查询

默认优先看：

- 7 天
- 30 天
- 90 天

适合判断：

- 是短期波动，还是持续变化
- 是突然起量，还是长期稳定
- 是已经掉下来了，还是还在上升

## 推荐调用顺序

真实数据查询时，优先顺序建议这样走：

1. 有 ASIN 或链接，先查 ASIN / Listing
2. 没有 ASIN，但有明确关键词，先查关键词
3. 用户问大方向，再查类目或子类目
4. 最后补趋势窗口，判断是短期还是长期

## 不该怎么做

- 不要为了凑结果，换成别的 Amazon 数据源
- 不要在 `proboost-Amazon-mcp` 不可用时自己脑补数字
- 不要把“接口类型”误写成“已经确认存在的具体函数名”

## 输出时怎么说

拿到数据后，不要把 MCP 当成主角。

不要写成：

- 调用了某某接口
- 返回了某某字段

更自然的说法是：

- 近 30 天这个词还有需求，但竞争已经不算低了
- 这个 ASIN 不是没量，问题是评论门槛已经比较高
- 这个类目最近不是明显上升盘，更像稳定盘

## 如果数据不全

如果某些字段没返回，直接说清楚：

- 当前只拿到价格和评论，销量数据没返回
- 趋势数据不完整，这个判断只能先做初筛
- 目前只能看个大概，不建议直接下最终结论
